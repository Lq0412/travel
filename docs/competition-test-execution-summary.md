## 五、核心接口耗时与连通性分析

测试方式：单机本地环境下，每个接口串行请求 5 次，记录平均响应时间、最大响应时间和最小响应时间。

| 接口 | 平均响应时间(ms) | 最大响应时间(ms) | 最小响应时间(ms) |
| --- | ---: | ---: | ---: |
| `GET /ai/monitor/health` | 9.0 | 28 | 4 |
| `GET /ai/trips/my` | 32.2 | 49 | 26 |
| `GET /ai/trips/{id}` | 30.8 | 51 | 25 |
| `GET /products/{id}` | 33.2 | 50 | 28 |

结论：

1. 本地基础查询接口平均响应时间稳定在 40 ms 以内，单次样本最大值为 51 ms。
2. 健康检查接口响应最快，适合作为监控探针。
3. 当前结果主要用于验证核心接口连通性与基础耗时，不等同于正式压测结果。
4. 当前结果只代表单机开发环境表现，不代表正式部署环境极限性能。

## 六、本轮修复的问题

### 1. 成功响应消息丢失

修复文件：

- `travel_backend/src/main/java/com/lq/travel/common/ResponseUtils.java`
- `travel_backend/src/main/java/com/lq/travel/common/ResponseDTO.java`

问题描述：

自定义成功消息参数被错误覆盖为固定值 `success`，导致接口层成功文案失真。

### 2. 商品表初始化 SQL 不兼容 MySQL

修复文件：

- `travel_backend/src/main/java/com/lq/travel/config/ProductSchemaInitializer.java`

问题描述：

启动阶段使用了 `ALTER TABLE ... ADD COLUMN IF NOT EXISTS ...`，当前 MySQL 环境不支持，导致后端无法启动。已改为先查询 `information_schema.columns` 再按需补列。

## 七、当前残余风险

1. 前端构建产物存在较大 chunk 告警，但不影响当前功能测试通过。
2. 后端测试和启动日志中存在 Mockito 动态 agent 警告，不影响当前测试结果，但未来 JDK 策略变化后需要调整构建配置。
3. AI 首字响应时间 TTFT 尚未形成独立量化基线。
