# 比赛测试体系设计

**目标**

为“智能 AI 旅游推荐平台”建立一套可执行、可证明、可回填比赛报告的数据化测试体系，覆盖自动化测试、手工测试、接口验证与基础性能验证。

**范围**

本轮测试仅覆盖比赛主链路中的 5 个模块：

1. 登录 / 注册
2. AI 工作台行程生成与连续追问
3. 行程保存 / 我的行程 / 行程详情
4. AI 监控台
5. 商品推荐 / 商品详情

**不纳入本轮范围**

1. 灵感广场
2. 社区论坛
3. 照片管理独立面板
4. 完整安全渗透测试
5. 超大规模压力测试

## 一、测试策略

### 1. 自动化测试

自动化测试分为前端和后端两层：

- 前端使用 `Vitest`
- 后端使用 `JUnit 5 + Mockito`

自动化测试目标不是覆盖全部页面 DOM，而是优先覆盖比赛主链路中的稳定业务规则、状态流转、数据解析、推荐逻辑和核心服务行为。

### 2. 手工测试

手工测试覆盖用户真实操作路径，用于补足自动化测试难以完整覆盖的页面联调、交互反馈、跳转链路与环境依赖行为。

### 3. 接口测试

接口测试以主链路 API 为核心，验证：

1. 正常请求
2. 参数缺失
3. 非法参数
4. 返回结构正确性
5. 关键失败提示

### 4. 性能测试

性能测试只做比赛可交付的基础验证，不做生产级压测。重点关注：

1. 登录接口响应
2. 行程查询 / 详情接口响应
3. AI 监控查询接口响应

AI 流式生成接口在本轮中只验证可用性和单次耗时，不做高并发压测；若执行时依赖外部模型服务，则在报告中明确标注环境边界。

## 二、自动化测试设计

### 1. 前端自动化测试设计

#### 1.1 登录 / 注册

- 校验空输入、密码不一致等前置校验逻辑
- 校验成功 / 失败时的页面跳转或提示分支

#### 1.2 AI 工作台

- 校验会话标题生成、历史会话摘要、结构化行程恢复
- 校验结构化流式结果解析
- 校验最近行程触发推荐文案与推荐卡片逻辑

#### 1.3 行程模块

- 校验结构化详情解析、按天过滤、摘要卡片生成
- 校验列表搜索、状态归一化、展示辅助逻辑

#### 1.4 AI 监控台

- 校验健康状态颜色、向量数量提取、任务状态摘要和同步结果摘要

#### 1.5 商品模块

- 校验商品筛选、详情读取、价格标签、下单记录保存逻辑

### 2. 后端自动化测试设计

#### 2.1 用户模块

- 注册请求关键字段校验
- 登录请求关键字段校验

#### 2.2 行程模块

- 结构化行程提取与恢复
- 行程保存时关键字段落库行为
- 行程详情关键读取逻辑

#### 2.3 AI 监控模块

- 状态查询 / 任务状态转换 / 结果摘要相关服务与工具行为

#### 2.4 商品模块

- 商品保存、地址解析、距离与展示相关逻辑

## 三、手工测试设计

每个模块提供独立用例，字段统一为：

1. 用例编号
2. 模块
3. 测试标题
4. 前置条件
5. 操作步骤
6. 预期结果
7. 优先级
8. 执行结果
9. 备注

## 四、数据采集要求

执行测试后，统一采集以下数据用于比赛报告：

1. 自动化测试总数、通过数、失败数
2. 手工测试用例总数、通过数、失败数、阻塞数
3. 接口测试总数、通过率
4. 性能测试平均响应时间、最大响应时间、错误率
5. 缺陷总数与严重级别分布

## 五、输出物

本轮测试应产出以下成果：

1. 手工测试用例文档
2. 前端自动化测试代码
3. 后端自动化测试代码
4. 测试执行结果记录
5. 比赛提交版测试报告

## 六、执行顺序

1. 编写测试设计与手工用例
2. 按 TDD 补充自动化测试
3. 执行前端与后端自动化测试
4. 在具备环境条件时执行接口和基础性能测试
5. 回填比赛测试报告
