# 行程详情页结构化展示设计说明

## 目标

当前“我的行程”列表点击进入详情后，只展示基础信息、每日亮点和照片管理，无法承接 AI 生成的完整结构化行程。此次改造的目标是把详情页升级为面向执行和回看的行程工作区。

改造后页面需要完成以下三件事：

1. 用户点击某个行程卡片后，可以直接看到完整的结构化行程内容，而不是只有摘要。
2. 结构化行程与地图联动展示，形成“看时间线 + 看路线”的详情体验。
3. 页面保持现有产品体系和 Ant Design 风格，不做营销页化或过度装饰化视觉。

## 设计结论

详情页采用“双栏执行工作区”方案：

- 顶部为行程基础信息和操作区。
- 内容区左侧为按天切换的时间线卡片列表。
- 内容区右侧为地图与点位路线展示。
- 时间线和地图上方增加结构化摘要卡片，用于展示天气、交通、住宿和预算。
- 原有照片管理能力保留为单独 tab，不与结构化详情混在同一视觉层级。

这是一个以结构化数据为主、照片和状态管理为辅的详情页方案。

## 页面结构

### 1. 顶部信息区

顶部信息区承接当前详情页已有能力，但信息组织要更明确。

展示内容：

- 目的地名称
- 行程天数
- 主题
- 出发日期
- 最近更新时间
- 行程状态

操作区保留：

- 返回我的行程
- 标记已完成
- 切换到照片管理

顶部区域不承担结构化正文展示职责，只负责建立上下文和主操作入口。

### 2. 结构化摘要区

在主内容区上方增加一行摘要卡片，用于概括结构化数据中的全局信息。

摘要卡片建议包含：

- 天气概览 `weatherOverview`
- 交通概览 `transportationOverview`
- 住宿概览 `accommodationOverview`
- 总预算或预计花费 `totalEstimatedCost`

如果某个字段为空，则不强行渲染空卡片，避免页面出现无意义占位。

### 3. 天数切换区

结构化行程通常存在多天数据。详情页默认展示单天视角，以提高可读性和执行感。

展示规则：

- 默认选中第 1 天。
- 顶部提供 `Day 1 / Day 2 / Day 3` 切换按钮。
- 额外提供“全部行程”视图作为补充，但不作为默认视角。

推荐原因：

- 更符合参考图中“当前一天执行面板”的使用方式。
- 左侧时间线密度更可控。
- 右侧地图路径也更容易聚焦，不会一次塞满所有点位。

### 4. 左侧时间线区

左侧是详情页的核心内容，使用结构化数据中的 `dailyPlans[].activities[]` 生成时间线。

单个活动卡片展示以下字段：

- 时间 `activity.time`
- 活动名称 `activity.name`
- 活动描述 `activity.description`
- 活动类型 `activity.type`
- 预计花费 `activity.estimatedCost`
- 建议提示 `activity.tips`
- 活动配图 `activity.imageUrl`（有则展示，无则使用占位）

时间线规则：

- 以活动时间为主轴展示。
- 时间点在左，卡片在右，形成纵向节奏。
- 餐饮活动和交通活动也纳入时间线，而不是单独拆出去。
- 如果当天存在 `meals`，优先与时间点合并展示；无法自然合并时作为辅助信息块放在当天底部。

视觉风格遵循企业级产品标准：

- 白底卡片
- 统一圆角
- 轻描边和轻阴影
- 清晰的层级和留白

不直接照搬参考图中的强毛玻璃和高模糊背景。

### 5. 右侧地图区

右侧地图复用现有 `DynamicMap.vue` 能力，但切换为详情页上下文使用。

地图行为：

- 默认显示当前选中天的数据。
- 根据活动位置绘制 marker 和连线。
- 当前选中活动高亮对应 marker。
- 如果活动没有坐标，沿用现有地理解析逻辑进行目的地和 POI 推断。

地图区域采用 sticky 固定，使左侧长时间线滚动时右侧地图仍保持可见。

### 6. 时间线与地图联动

联动是详情页体验的关键增强项，但应分阶段实现。

第一阶段联动：

- 切换天数时，左侧列表和右侧地图同步切换。
- 点击左侧活动卡，右侧地图高亮对应点位。

第二阶段联动：

- 点击地图 marker，左侧滚动并定位到对应活动卡。
- 地图路线支持 hover 或 current 状态强调。

第一阶段足以满足当前目标，第二阶段作为增强。

### 7. 照片管理区

照片管理保留为独立 tab，不与结构化正文并排混杂。

tab 规则：

- `概览` tab：承载结构化摘要、天数切换、时间线和地图。
- `照片` tab：保留当前 `TripPhotoManager`。

这样可以避免详情页同时承担“结构化阅读”和“素材管理”两种不同任务。

## 数据流设计

### 1. 数据来源

详情页主数据仍来自 `getTripById`。

关键变化是从 `trip.structuredData` 解析出完整 `StructuredItinerary`，而不是仅依赖 `trip.dailyHighlights`。

数据优先级：

1. `trip.structuredData` 解析成功时，以完整结构化行程为准。
2. 若 `structuredData` 缺失或解析失败，则退回当前 `dailyHighlights` 摘要视图。
3. 照片数据仍继续使用 `trip.photos`。

### 2. 前端解析层

建议在详情页中引入一个明确的结构化解析步骤：

- 读取 `trip.structuredData`
- 安全 `JSON.parse`
- 校验是否符合 `StructuredItinerary` 基本结构
- 得到 `parsedItinerary`

不要在模板中直接解析字符串，也不要把解析逻辑散落在多个 computed 中。

如有必要，可将解析逻辑抽到独立工具函数，供工作台和详情页复用。

### 3. 展示派生数据

详情页内部需要额外派生以下数据：

- 当前选中天数 `selectedDay`
- 当前天的活动列表
- 当前地图展示路径
- 当前选中活动 `selectedActivityKey`
- 摘要卡片列表

这些派生状态应以 `parsedItinerary` 为唯一数据源，避免与 `dailyHighlights` 双轨并存造成冲突。

## 组件拆分建议

为避免 `TripDetailPage.vue` 继续膨胀，建议拆分为以下组件：

- `TripDetailSummaryCards.vue`
  - 负责天气、交通、住宿、预算摘要展示
- `TripDaySwitcher.vue`
  - 负责天数切换和“全部行程”切换
- `TripItineraryTimeline.vue`
  - 负责左侧时间线和活动卡片
- `TripItineraryMapPanel.vue`
  - 包装地图容器，并把详情页所需状态传给 `DynamicMap`

`TripDetailPage.vue` 只负责：

- 拉取详情数据
- 解析结构化行程
- 管理 tab / selectedDay / selectedActivity
- 编排整体布局

## 状态与异常处理

需要覆盖以下情况：

1. 行程存在但没有 `structuredData`
   - 显示当前基础概览和 `dailyHighlights`
   - 明确提示“该行程暂无结构化日程”

2. `structuredData` 解析失败
   - 页面不崩溃
   - 控制台保留错误信息
   - 前端提示“结构化数据解析失败，已展示基础信息”

3. 某些活动没有坐标或图片
   - 地图使用回退定位策略
   - 活动卡使用默认占位

4. 多天数据和 `days` 字段不一致
   - 以前端归一化后的 `dailyPlans.length` 为展示依据

## 响应式策略

桌面端是主要场景，按双栏布局实现。

移动端规则：

- 地图区域下沉到时间线下方
- 天数切换保留横向滚动
- 摘要卡片改为单列或双列
- 活动卡片保持纵向流式排列

移动端不强求桌面端的 sticky 地图体验，但要保证可读和可操作。

## 实现分期

### 第一阶段

- 解析 `trip.structuredData`
- 详情页加入结构化摘要区
- 实现按天切换
- 左侧展示结构化时间线
- 右侧接入地图并按天切换

### 第二阶段

- 左右联动高亮
- 活动配图增强
- 全部行程视图优化
- AI 微调入口

## 与现有代码的衔接

当前项目已经具备以下基础：

- `TripDetailPage.vue`：详情页容器和基础信息
- `DynamicMap.vue`：结构化地图展示能力
- `StructuredItinerary` 类型定义
- 工作台中对结构化数据的处理经验

因此本次改造不需要重新发明地图或结构化类型，只需要把工作台已有结构化展示能力收敛到详情页场景，并做页面层级重组。

## 成功标准

改造完成后应达到以下标准：

1. 用户从“我的行程”点击任一行程后，能立即看到结构化日程，而不是只有摘要。
2. 页面默认展示单天执行视角，阅读成本低。
3. 地图与时间线能围绕同一份结构化数据同步展示。
4. 没有结构化数据时页面能优雅降级，不出现空白或报错。
5. 详情页视觉风格与现有项目一致，并符合 Ant Design 风格下的企业级产品体验。
