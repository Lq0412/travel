# 智能 AI 旅游推荐平台作品介绍文档（源码更新版）

## 一、项目概述

### 1.1 项目定位

本项目面向“个人旅行规划 + 行程资产沉淀”场景，构建了一套基于 Web 的智能 AI 旅游推荐平台。用户可以通过自然语言输入出行需求，由系统完成会话创建、流式生成、结构化提取、地图可视化、行程保存与详情回看，形成从“提出需求”到“沉淀个人旅行资产”的完整闭环。

与传统旅游攻略网站或单轮 AI 问答页面不同，本项目强调三点：

1. AI 输出不仅是文本，还会被提取为结构化行程数据。
2. 结构化结果会同步映射到时间轴和地图，实现可视化理解。
3. 生成后的结果可以沉淀到“我的行程”中，形成可持续管理的个人内容资产。

### 1.2 当前源码对应的产品形态

结合最新源码，当前系统已经形成“前台体验链路 + 后台运行证明链路 + 辅助扩展模块”的完整产品结构：

1. 首页 `HomePage`
2. AI 旅行工作台 `WorkspacePage`
3. 我的行程列表 `TripsPage`
4. 行程详情页 `TripDetailPage`
5. 灵感广场 `ForumPage`
6. 个人中心 `ProfilePage`
7. 商品库 / 商城页 `MallPage`
8. AI 监控中心 `AIAdminMonitorPage`
9. 用户登录 / 注册页

其中，比赛主展示链路仍然以“首页 -> 工作台 -> 我的行程 -> 行程详情 -> AI 监控中心”为核心；灵感广场、个人中心和商品库则体现了平台继续扩展为完整个人服务产品的能力。

### 1.3 项目建设目标

本项目当前版本的建设目标主要包括：

1. 提供低门槛的自然语言旅游规划入口。
2. 将大模型输出稳定转化为结构化行程资产。
3. 实现时间轴与地图联动，提升行程结果的可读性和可验证性。
4. 打通会话、保存、回看、筛选、详情查看等个人中心链路。
5. 建立监控、知识补齐、Milvus 同步等后台证明能力，增强答辩可信度。

## 二、项目实际功能与使用方式

### 2.1 用户侧已实现功能

根据当前前后端源码，系统已实现如下能力：

1. 用户注册、登录、退出、获取当前用户信息。
2. 个人资料编辑、头像上传、基础账户信息维护。
3. AI 工作台自然语言提问与连续追问。
4. 会话创建、历史会话切换、消息历史恢复。
5. SSE 流式输出 AI 内容。
6. 从流式结果中提取结构化行程 JSON，并在前端进行标准化展示。
7. 行程时间轴展示与差异更新提示。
8. 高德地图联动展示、点位解析与路线渲染。
9. 当前行程保存到“我的行程”。
10. 行程列表检索与详情查看。
11. 行程完成状态标记。
12. 最近行程推荐触发“城市美食商品推荐卡片”。
13. 商品库本地维护，用于工作台推荐卡片的数据源。
14. 灵感广场静态灵感模板展示。
15. AI 监控中心的健康检查、Milvus 查询、知识补齐任务提交、任务状态轮询与入库同步。

### 2.2 使用流程

当前版本的典型使用方式如下：

1. 用户进入首页，可直接从首页 Hero 区带着预设 prompt 跳转到工作台。
2. 用户登录后进入 AI 工作台，通过自然语言描述目的地、天数、预算和玩法偏好。
3. 系统创建会话，调用后端流式 AI 接口返回连续内容。
4. 前端在接收流式内容的同时，提取结构化行程数据并渲染时间轴。
5. 若存在可定位地址，地图会同步完成 POI 检索、地理编码和路径展示。
6. 用户可以在已有会话上继续追问，系统保留上下文并生成更新后的行程。
7. 用户可将当前结构化行程保存到“我的行程”。
8. 在“我的行程”中，可查看历史记录、搜索目的地或主题，并进入详情页。
9. 在 AI 监控页中，可查看 Milvus 健康状态、知识向量规模、知识补齐任务和同步结果。

### 2.3 当前版本的边界说明

为保证文档与源码一致，需要说明以下几点：

1. 灵感广场当前为静态灵感承接页，尚未接入真实社区发布流和详情接口。
2. 商城页当前使用前端本地商品库数据，主要服务于工作台中的城市美食推荐展示。
3. 行程照片管理后端接口与独立前端组件已具备，但最新详情页主视图当前重点展示时间线和地图，未在主页面中完整挂载照片管理面板。
4. AI 工作台的“去看看 / 购买”按钮目前保留为展示能力，尚未接入真实交易闭环。

## 三、项目亮点

### 3.1 从 AI 文本生成升级为结构化行程生产

本项目最大的亮点，不是“能回答旅行问题”，而是能把大模型输出转换为包含 `destination`、`days`、`budget`、`theme`、`dailyPlans`、`totalEstimatedCost`、`tips` 等字段的结构化对象，并驱动前端页面直接消费。这使 AI 结果真正进入业务链路，而不是停留在一次性文本展示层面。

### 3.2 流式生成、结构化提取与页面联动并行

系统通过 SSE 流式接口返回模型内容，前端在渲染聊天记录的同时抽取结构化标记，并将其同步映射到时间轴和地图。这种“边生成、边解析、边展示”的能力，既改善了等待体验，也增强了答辩演示时的技术观感。

### 3.3 多智能体协同提示编排

后端引入了多角色协同策略，在提示中拆分“需求分析 Agent、知识 Agent、路线设计 Agent、质检 Agent”等职责，对天数、预算、动线、字段完整性和结果可执行性进行约束。相比单一 Prompt 的直接生成，这种方案更能体现项目的 AI 应用设计能力。

### 3.4 RAG 检索增强与双路回退

系统优先通过 Milvus 完成向量检索；若召回不足，再回退至本地知识表进行关键词检索，并进行去重与排序。该机制降低了旅游场景中景点、美食、经验信息的幻觉风险，提高了生成方案的事实支撑能力。

### 3.5 地图联动不只是“打点”，而是“可走的路线”

地图模块综合使用高德 JSAPI 的 POI 搜索、地理编码、驾车路径规划、信息窗体与视口自适应能力，支持按天分色、同点偏移、无坐标兜底和目的地中心回退。相比普通地图标点，本项目更强调真实路线表达与空间逻辑可视化。

### 3.6 从单次生成扩展为个人中心资产闭环

用户生成的行程可以保存到个人中心，在“我的行程”列表中继续查找、查看和标记完成。该闭环能力符合“个人中心”类赛题对于内容沉淀、内容回看和长期管理的要求。

### 3.7 引入商品推荐联动，增强商业扩展潜力

当前版本新增了“最近行程 -> 城市商品匹配 -> 聊天卡片推荐”的联动逻辑，并配有独立商品库页面。虽然交易能力尚未真正接入，但这一设计已经体现出平台向“规划 + 推荐 + 消费转化”扩展的产品可能性。

### 3.8 具备后台监控与知识补齐流水线

AI 监控中心不仅展示健康状态和向量规模，还提供知识补齐任务下发、任务状态追踪、自动入库同步和 Milvus 查询结果展示。这使作品能够同时提供“用户前台可用性”与“后台工程可证明性”。

## 四、Web 网页使用情况概述

### 4.1 首页

首页当前作为平台品牌入口与首屏转化页面存在，采用大图 Hero 形式承载热门目的地切换和“开始规划”入口。用户可以带着预设目的地 prompt 直接跳转工作台，降低首次使用门槛。

### 4.2 AI 工作台

工作台是项目核心页面，当前包含以下关键区域：

1. 顶部会话切换、新建会话、保存行程、地图开关、我的行程入口。
2. 中部时间轴与地图双栏主舞台。
3. 右上悬浮聊天记录面板。
4. 底部最近行程卡片与输入区。

该页面支持会话恢复、结构化行程回填、流式文本展示、地图联动和最近行程触发商品推荐，是比赛现场最重要的演示页面。

### 4.3 我的行程

“我的行程”页面使用卡片网格展示历史行程，支持关键词搜索，卡片内展示目的地、天数、日期与状态，并通过首张照片或自动图像补全作为封面。页面承担的是个人中心资产库角色。

### 4.4 行程详情页

行程详情页当前版本采用“左侧时间线 + 右侧沉浸式地图”的布局，支持按天切换、活动选中联动地图聚焦、统计摘要查看和行程状态完成标记。该页重点体现“保存后仍可继续阅读和使用”的产品逻辑。

### 4.5 灵感广场

灵感广场页面当前提供静态精选路线模板和主题标签，定位为“帮助用户组织需求、再跳转工作台”的辅助页面。虽然尚未接入真实社区流，但其存在说明项目已开始考虑内容生态入口。

### 4.6 个人中心

个人中心当前支持用户信息查看、资料编辑、头像上传和密码修改弹窗等功能，体现了项目已具备基础账号体系和个人资料维护能力。

### 4.7 商品库 / 商城页

商城页目前维护自定义美食商品库，支持新增、编辑、删除、筛选和推荐状态切换。该模块虽然是本地数据驱动，但与工作台推荐卡片存在联动，能够作为后续商业化方向的原型验证。

### 4.8 AI 监控中心

AI 监控中心采用“摘要卡片 + 证据条 + 双栏面板”的结构，左侧负责新建知识补齐流水线，右侧负责本机任务记录追踪，是本项目最适合用于答辩中证明后台能力的页面。

## 五、项目涉及的关键技术

### 5.1 前端技术体系

前端采用 Vue 3、TypeScript、Vite、Pinia、Vue Router 和 Ant Design Vue 构建。页面逻辑使用 Composition API 组织，前端工程目录按照 `pages`、`components`、`composables`、`api`、`utils`、`types` 等维度拆分，具备较清晰的工程结构。

### 5.2 后端技术体系

后端采用 Spring Boot 3、Java 21、MyBatis-Plus、MySQL、Redis、RabbitMQ、Spring Session、Resilience4j、Knife4j 等技术栈，能够支撑用户体系、AI 接口、会话持久化、知识任务、监控接口和基础工程保障能力。

### 5.3 SSE 流式交互

后端通过 `SseEmitter` 提供 AI 流式接口，前端使用 `fetch + ReadableStream` 持续读取事件流，并对 `start / data / result / complete / error` 等事件进行解析，保证交互过程具备实时反馈。

### 5.4 结构化行程提取与容错标准化

后端 `StructuredItineraryExtractor` 能够从普通文本或代码块中定位 JSON，处理单引号、尾逗号、未加引号字段、中文标点等常见异常，并统一规范 `dailyPlans`、`activities`、`location`、`estimatedCost` 等字段，是本项目结构化落地的关键技术基础。

### 5.5 会话管理与上下文恢复

系统支持会话创建、消息持久化、历史消息查询、最近消息截断加载以及前端结构化行程恢复。用户可以在旧会话上继续规划，而不是每次重新开始，这使平台更接近真实使用场景。

### 5.6 多智能体协同提示工程

后端 `TravelMultiAgentCoordinator` 会根据任务文本推断目的地，拼装多角色协同 Prompt，并注入目的地线索、RAG 检索结果和路线设计硬规则，提升规划结果的稳定性、完整性和可执行性。

### 5.7 RAG 检索增强

后端 `TravelRagServiceImpl` 采用“Milvus 向量召回优先、本地知识表关键词检索回退”的双路方案，对景点、美食和经验内容进行融合召回和去重排序，为模型生成提供事实参考上下文。

### 5.8 地图与位置服务

前端通过高德 JSAPI Loader 动态加载地图，结合 Geocoder、PlaceSearch、Driving、Marker、Polyline、InfoWindow 等能力完成点位定位和路线规划。后端还会通过高德天气接口补充目的地近期天气信息，用于辅助路线设计。

### 5.9 异步任务与知识补齐流水线

后台知识补齐任务通过 `AiTask` 表、RabbitMQ 消息队列和任务状态轮询机制实现，支持业务键幂等、失败重试、成功后 Milvus 入库同步，体现出后台任务治理能力。

### 5.10 登录态、限流与配额控制

系统使用会话登录态校验、接口鉴权拦截、Resilience4j 限流与 Token 配额控制来约束 AI 能力调用频次，避免滥用并增强系统稳定性。
